# Dev Data Reset

Reset disposable dev data while preserving Clerk-backed accounts and any shared tenant that another active user still uses, then republish and reinstall the four first-party plugins. Load `convex-admin-ops` for CLI mechanics (Windows invocation, JSON args, readback interpretation), `plugin-system` for publish/release invariants, and `app-playwriter-harness` for the browser reseed phase.

## What survives a reset

- Every `users` doc with a non-null `clerkUserId`, its Clerk auth state, profile, Polar billing/customer state, and usable default tenant (org + workspace). This is a hard reset invariant: never use a mode that deletes one of these user docs.
- Shared organizations and workspaces that still have another active preserved user. Their shared content remains because the existing per-user reset does not destroy a tenant that another user still uses.
- Convex deployment env vars (including `PLUGIN_SECRETS_ENCRYPTION_KEY` — encrypted secrets are wiped, not the key).
- Disposable personal content, empty organizations, plugin registry data (versions, reviews, claims, publisher secrets, R2 artifacts), installations and installation secrets, event runs, and anonymous users (including their `users` docs) are deleted.

## Preflight

1. Confirm the target from `packages/app/.env.local` (`CONVEX_DEPLOYMENT`, currently `dev:grand-finch-267`). Dev only — never `--prod`.
2. **Back up before wiping.** Write the `convex export --path <backup.zip>` snapshot under `../t3-chat-+personal/+ai/dev-data-reset-YYYY-MM-DD/`, not inside the repository or the OS temp directory. It is rollback material, not a source of values to print.
3. **Confirm recovery inputs.** Confirm the deployment already has `GITHUB_TOKEN_IMPORT`, `OPENAI_API_KEY`, `MODAL_TOKEN`, `MISTRAL_API_KEY`, `MODAL_MEDIA_AUDIO_URL`, and `MODAL_FILE_CONVERTER_URL`. Publishing needs the first two; plugin publisher secrets use the remaining mapped values below. Never run a command that prints secret values into captured output, and never echo, log, or paste them into a report.
4. If a value is missing, stop before the wipe. Use the provider's supported credential issuance or rotation flow, or ask Ray to provide the value directly, then store it in the Convex deployment environment. Never use Playwriter, DOM reads, screenshots, transcripts, or service pages to extract an existing token. Missing Modal endpoint URLs may be regenerated by deploying `packages/app/modal/files_markitdown.py` with `packages/app/scripts/modal-cli.ps1` and `plugins/bonobo-plugin-video/modal/media_audio_extractor.py` with that plugin's `scripts/modal-cli.ps1`; record only the resulting endpoint URL in Convex env.

## Phase 1 — Wipe (Convex CLI)

Run from `packages/app` with the direct Node CLI invocation (`vp env exec node node_modules/convex/bin/main.js run --typecheck disable --codegen disable ...`); see `convex-admin-ops` for arg-quoting hazards.

Before a breaking schema deploy, clean old plugin rows with the currently deployed code. Stop publishing and export every existing cleanup attempt id and `cleanupAt`. Wait until the latest grace deadline, then run the deployed `plugins:run_publish_artifact_cleanup_attempt` for each id until readback shows `plugins_publish_artifact_cleanup_attempts` is empty. Next export the plugin names and run the deployed `preview_hard_delete_registered_plugin` → `hard_delete_registered_plugin_now` → preview readback flow until `plugins_versions` is empty. Only then deploy the strict current schema that requires `uploadId`. This is a dev-data wipe, not a compatibility migration; do not make the new field optional to carry old rows through deployment.

1. **Plugin registry.** Stop publishing first; no publish action may still be running while destructive cleanup and readback execute. Export `plugins_versions`, `plugins_version_reviews`, and `plugins_publish_artifact_cleanup_attempts` with `--format jsonArray --limit 1000`, doubling a table's limit whenever its returned count equals the limit. Collect the distinct union of `plugins_versions.name`, `plugins_version_reviews.pluginName`, and `plugins_publish_artifact_cleanup_attempts.pluginName`. For each name run `plugins:preview_hard_delete_registered_plugin` → `plugins:hard_delete_registered_plugin_now` → preview readback. Every preview count must be zero; rerun the delete if it exhausts its 50-batch budget. The name-scoped flow deletes registered versions, rejected reviews without a version, interrupted-upload keys/attempts, installations, and version-owned run history.
2. **Publisher claims.** Export every `plugins_publisher_repositories._id` with the same exact JSON/doubling-limit rule. After all name previews are zero, call `plugins:hard_delete_publisher_repository_now` once for each id. This separate id-scoped step is required because a repository can be claimed before its manifest supplies a plugin name; it deletes the claim and its secrets idempotently.
3. **Users.** Record the reset start time. Export `users` with `data users --format jsonArray --limit 1000`; if the array contains exactly 1000 docs, double the limit until the returned count is below it. Record `_id` and `clerkUserId` for every doc, and record the completion time of each successful user/mode invocation. Process the two groups in this order so shared organizations stay attached to a preserved account:
   - First, successfully invoke every user with a `clerkUserId` using `{"userId":"<id>","purgeUserMod":"data"}`.
   - Then, successfully invoke every user without a `clerkUserId` using `{"userId":"<id>","purgeUserMod":"data_auth_and_user_record"}`.
   - One successful invocation is enough for bounded local work because it schedules its own continuations. If Clerk, Polar, or another external dependency makes the invocation fail, correct that problem and retry the same user and mode.
4. **Deletion queue.** Poll until `data_deletion_requests` is empty. Successful auth-removing calls already enqueue the Workpool. Do not enqueue a second worker while one is queued or running. Manually call `data_deletion:enqueue_deletion_requests_processing` only as recovery after logs and repeated readback show that the queue has stopped and no worker remains.
5. **Orphan sweep (conditional).** Plugin publishes materialize version-keyed source trees (`/<pluginVersionId>/...`) under the GLOBAL organization / PLUGINS workspace, and `plugins:hard_delete_registered_plugin_now` (step 1) already sweeps them per version. After step 1, count `files_nodes` for GLOBAL/PLUGINS — it should be zero; a nonzero count means a version doc was deleted outside the hard-delete flow, and the leftover trees can be drained with `plugins:delete_plugin_source_tree_batch` (`{"pluginVersionId":"<id>"}`, rerun until `done: true`). GitHub mirror mounts under GLOBAL/GITHUB are unrelated to plugins and are managed by `github_mounts`.
6. **Expired grants (conditional).** If `public_api_grants` shows a backlog of expired docs, run `public_api:cleanup_expired_grants_until_done` with `{}`.

The existing user deletion logic owns tenant cleanup. It deletes an organization only when no active user remains. If another user still belongs to it, the organization stays. Do not add a deployment-wide organization delete or special shared-organization path.

Readback gate before declaring the wipe done: inspect `users:hard_delete_user_now` scheduled rows created since this reset started. None may be `pending` or `inProgress`. A failed row is unresolved unless a later explicit invocation for that same user and mode succeeded; investigate and retry that pair, then require no later pending or running continuation. Older failed rows may remain visible for seven days and do not fail a later reset. `plugins_versions`, `plugins_version_reviews`, `plugins_publish_artifact_cleanup_attempts`, `plugins_publisher_repositories`, `plugins_publisher_repository_secrets`, `plugins_workspace_installations`, `plugins_workspace_installation_secrets`, `plugins_workspace_event_handlers`, `plugins_ui_sessions`, `plugins_event_runs`, `plugins_event_run_calls`, and `data_deletion_requests` are empty; `users` contains only Clerk-backed docs, each still with its original `clerkUserId`, profile, and Polar linkage. Repeat the scheduled-function and table readback once before reseeding. The second readback is the barrier that prevents a final bounded reset continuation from deleting newly seeded plugin data.

## Phase 2 — Reseed plugins (app UI via Playwriter)

Drive Ray's signed-in app tab (Clerk user preserved by the wipe). Reload the tab first, then navigate fresh from the app root. Mind two known frictions: the Convex dev cold start (first action can take ~60s+) and the `plugins_manage` rate limiter (token bucket, capacity 2, ~6/min — space plugin mutations ~15s apart).

Browser driving (verified 2026-07-06): Edge's own `--remote-debugging-port=9222` no longer binds on the default user-data-dir (Chromium M136+ restriction) — use the Playwriter CLI instead (`vp env exec pnpx playwriter session new`, then `vp env exec pnpx playwriter -s <id> -f <script.js>`). On this machine multi-line `-e` code is truncated at the first newline, so always use `-f` with a script file; each execute call has a ~10s cap, so keep calls to one action and verify state via `convex data` readbacks instead of in-page sleeps. If the app tab shows the "Something went wrong" error boundary, `page.reload()` and continue.

First-party plugin repositories (git submodules under `plugins/`):

- `https://github.com/raythurnvoid/bonobo-plugin-gallery`
- `https://github.com/raythurnvoid/bonobo-plugin-image`
- `https://github.com/raythurnvoid/bonobo-plugin-video`
- `https://github.com/raythurnvoid/bonobo-plugin-pdf`

For each plugin, in order:

1. **Claim + publish.** On `/w/:organizationName/:workspaceName/plugins/publisher`, paste the repository URL into the claim input (placeholder `https://github.com/owner/plugin-repo`), click `Claim`, then `Publish` on the repository card. Publishing imports the artifact from GitHub and runs the AI review — allow a few minutes; the card shows the last attempt outcome. A `rejected`/`failed` status message on the card is the triage starting point (review verdict vs GitHub fetch).
2. **Publisher secrets.** On the plugin detail page (`/w/:organizationName/:workspaceName/plugins/<name>`), restore the required values in the publisher Secrets section from trusted deployment values or explicit user-provided values:
   - `gallery`: no publisher secrets
   - `image`: `OPENAI_API_KEY` — origins `https://api.openai.com`
   - `video`: `MISTRAL_API_KEY` (`https://api.mistral.ai`), `OPENAI_API_KEY` (`https://api.openai.com`), `MODAL_MEDIA_AUDIO_URL` and `MODAL_TOKEN` (origin = the Modal extractor origin)
   - `pdf`: `MODAL_FILE_CONVERTER_URL` and `MODAL_TOKEN` (origin = the file-converter origin)
3. **Install.** On the same detail page, click `Install` and accept the consent modal (capabilities + outbound origins). The button reads `Update`/`Reinstall` if an installation already exists.
