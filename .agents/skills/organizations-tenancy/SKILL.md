# Mental model

- An **organization** groups **workspaces**. Each organization has exactly one **primary** (default) workspace referenced by `organizations.defaultWorkspaceId` and flagged `organizations_workspaces.default === true` for that doc.
- **Membership** is per **workspace**: table `organizations_workspaces_users` keys `(userId, organizationId, workspaceId)`. A user “sees” an organization in `list` if they have **any active** workspace membership under that organization (`active === true`).
- **Product rule (organization membership by primary workspace):** Organization-level operations that mean “is this user a member of this organization?” (e.g. `edit_organization`) require a membership doc on that organization’s **primary/default workspace** (`defaultWorkspaceId`), not merely on any workspace in the organization.
- **Organization ownership** lives on `organizations.ownerUserId`, a required `users` id. Access-control role assignments still represent role membership: `owner`, `admin`, and `member` assignments are scoped to a workspace, and the organization default workspace means organization-wide authority. Keep the default-workspace `owner` role assignment as a mirror for role display and ACL compatibility, not as the source of truth for the organization owner id.
- **Billing mode** lives on `organizations.billingMode`. Default/personal organizations always behave as `"user"` billing. Non-personal organizations are created with `"user"` billing by default and the organization owner may switch them to `"organization_owner"`.
- `organizations.ownerUserId` is the billing payer only when `organizations.billingMode === "organization_owner"`. In `"user"` mode, the actor/member remains the payer.
- Ownership transfer changes future owner-billed usage only. It does not rewrite historical usage, move snapshots, transfer credits, or affect `"user"` mode billing.
- Tenant-scoped APIs should take `membershipId` whenever the caller is operating inside a current organization/workspace. Derive `organizationId` and `workspaceId` from the active membership doc instead of trusting client-provided scope strings. Comment APIs (`chat_messages.*`) follow this rule for create/add/archive/list/get.
- The **public API** (`api.organizations.*` mutations/queries and Convex helpers they call) is the contract. The **database schema** is wider (optional fields, flags) so migrations and edge docs can exist; do not assume “schema allows it ⇒ product allows it.” Enforce invariants in Convex handlers and in `packages/app/convex/organizations.ts`.

# Comment terminology

- Prefer **doc/docs** for Convex table entries in comments and guidance. Avoid **row/rows** unless quoting an API field or database-neutral source text.
- Use **default tenant** for the user’s default `personal` organization plus default `home` workspace.
- Use **default-workspace membership doc** for the `organizations_workspaces_users` doc on an organization’s primary/default workspace. Avoid vague wording like “home membership.”
- Use **default organization/workspace docs**, **workspace doc and related structure**, or **organization doc and related structure** instead of “shell” in explanatory comments.
- Use **usable account after reset** or spell out preserved `users` doc/auth/profile/billing state instead of “live account.”
- Use **reset-owned data** for data that the admin data reset is allowed to purge.
- Treat missing or inconsistent default tenant state as an invariant error in reset/delete comments. Do not describe it as a repair path unless the code intentionally repairs it.

# Default tenant (per user)

- On **user bootstrap** (anonymous create, Clerk resolve/link), the app ensures a **default organization + default workspace** for that user via `organizations_db_ensure_default_organization_and_workspace_for_user` in `packages/app/convex/organizations.ts` (called from `packages/app/convex/users.ts`).
- **Stored names** (normalized slugs): organization `personal`, workspace `home` (see `DEFAULT_ORGANIZATION_NAME` / `DEFAULT_WORKSPACE_NAME` in `packages/app/convex/organizations.ts`). UI may display title case; API/storage uses these slugs.
- **User doc cache:** `users.defaultOrganizationId` and `users.defaultWorkspaceId` point at that default tenant. If the organization pointer is absent or points to a missing organization doc, `organizations_db_ensure_default_organization_and_workspace_for_user` creates a new `personal`/`home` tenant and rewrites both pointers. If the pointed organization exists, the helper returns without validating or repairing the workspace pointer or membership docs.
- **Exactly one default tenant per user** in normal flows: the UI does not create a second default organization; internal `organizations_db_create(..., default: true)` is for provisioning that tenant. Non-default organizations are created with `default: false`.
- Default organizations use `billingMode: "user"` and do not expose organization billing management.
- The default `personal` organization remains private. Invitation/member-management mutations must reject it, and the main nav hides the Users entry for default organizations even though the Users page may render for direct/debug URLs.

# Anonymous upgrade preserves tenancy and data

- When `resolve_user` receives an `anonymousUserToken`, it upgrades the **same** Convex `users` doc to a Clerk-linked user in `packages/app/convex/users.ts`.
- The anonymous user record remains the canonical user record after upgrade. The code patches `clerkUserId` onto that same doc instead of creating a replacement user for the upgraded account.
- Because the same user doc is preserved, the user keeps the same `defaultOrganizationId` and `defaultWorkspaceId` during the normal upgrade path.
- The user therefore keeps the same organization/workspace memberships and the same organization/workspace-scoped data already attached to that identity and tenant.
- Treat "anonymous user upgrades to signed-in user and keeps the same personal organization/home workspace and their data" as a product invariant, not a best-effort repair flow.

# Every new organization gets a Home workspace

- `organizations_db_create` always inserts a default workspace named `home` (`default: true`) and links `organizations.defaultWorkspaceId`.
- **`create_organization`** (public) calls `organizations_db_create` **without** `default: true`, so it creates a **non-default** organization plus its `home` workspace. That organization is not the user’s `personal` default.
- `organizations_db_create` stores `organizations.ownerUserId = userId`, creates the mirrored owner assignment on the default workspace, seeds default access-control grants, and stores `billingMode: "user"`. This applies to both default and non-default organizations, but only non-default ownership consumes the user’s `extra_organizations` quota.

# Access control model

Canonical access-control details live in `../access-control/SKILL.md`.

- Roles are `owner`, `admin`, and `member`.
- The single effective organization owner is `organizations.ownerUserId`. A matching `role: "owner"` assignment on `organization.defaultWorkspaceId` is maintained for role display and ACL compatibility. Owner is a system role with full organization authority.
- `admin` and `member` authority comes from docs in `access_control_permission_grants`. Current seeded member grants intentionally preserve broad collaborator behavior except member management; future tightening should change grants/checks rather than table shape.
- Non-default workspace role assignments are local to that workspace. Default-workspace role assignments act as organization-wide fallback.
- ACL grants support `principalKind: "role"`, `"user"`, and `"public"` for resource kinds `organization`, `workspace`, `file`, and `thread`. `resourceId` is stored as a stringified Convex id; owning mutations/actions load the resource first and derive the access-control scope from that doc.
- Permission checks order: `organizations.ownerUserId`, direct user grant, public grant when explicitly allowed, target-workspace role grants, then default-workspace organization-wide role grants.
- Use `access_control.get_current_user_role({ organizationId, workspaceId })` for current-user UI role display at the requested scope, and `access_control.get_organization_workspace_user_role({ organizationId, workspaceId, userId })` when a UI needs one listed user's role at that same scope. The default/home workspace role view is the organization role view; non-default workspace views show workspace-local roles. Do not embed role dictionaries into `organizations.list`; keep role lookups cached by the concrete organization/workspace/user scope.

# Edit and delete rules

| Rule                                                                                                                                                | Enforcement (typical)                                                                                                                                                                                                                     |
| --------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Cannot **edit** the default organization (`organizations.default === true`)                                                                         | `edit_organization`                                                                                                                                                                                                                       |
| Cannot **edit** the default/primary workspace (`workspace.default` or `workspace._id === organization.defaultWorkspaceId`)                          | `edit_workspace`                                                                                                                                                                                                                          |
| Cannot **delete** the default organization                                                                                                          | `delete_organization`                                                                                                                                                                                                                     |
| Cannot **delete** the default workspace                                                                                                             | `delete_workspace`                                                                                                                                                                                                                        |
| Only the organization owner may **delete** a non-default organization                                                                               | `delete_organization` checks `organizations.ownerUserId`                                                                                                                                                                                  |
| Regular members keep broad current organization/workspace abilities except organization deletion, default-workspace deletion, and member management | Seeded `access_control_permission_grants` keep current broad behavior while reserving `organization.members.manage` and `workspace.members.manage` for admins/owners; `remove_user_from_organization` still lets members leave themselves |

**Refactoring note:** Only the **primary** default **workspace** is protected from edit across all organizations. Do **not** block editing **all** workspaces in the user’s default `personal` organization based solely on `organization.default`; only the **`home`** (primary) workspace is special.

**Known implementation mismatch:** `delete_workspace` currently accepts either `workspace.update` on the target workspace or `organization.update` on the organization. It does not enforce the declared and seeded `workspace.delete` permission. Product policy has not decided whether to use only `workspace.delete` or keep an organization-level fallback.

# Invitations / adding members

- **`invite_user_to_organization_workspace`:** accepts either `userIdToAdd` or an exact normalized email resolved from `users_anagraphics.by_email` (id wins when both are provided), rejects missing/deleted/current users, and rejects default organizations. A new invite adds membership and a `member` role assignment to the organization `home` workspace, adds them to the selected workspace when different, and creates one unread `notifications` doc. An idempotent repeated invite returns success without creating another notification.
- **Default organization:** if `organization.default`, mutations fail with `Cannot add user to default organization`. That blocks inviting into **personal** and into **any workspace under personal** (including `home`), matching “no collaborators on the default tenant.”
- **Users page:** `/w/$organizationName/$workspaceName/users` composes granular queries: `organizations.list` for the visible organization and workspaces, `organizations.list_organization_workspace_users` for the `home` roster plus per-workspace user ids, `users.get_anagraphic` for profile details, and access-control role queries for badges/actions. The organization `home` workspace remains the organization user roster, invite/remove/transfer actions stay disabled for `personal`, and the main sidebar hides the Users nav item whenever `organization.default === true`; direct URLs may still render this guarded/read-only page. `users.get_anagraphic` currently returns the full anagraphic doc, including email, without an auth or membership check. Treat these cross-user calls as a known privacy gap; do not add more until product policy defines the safe fields and membership scope.
- **Removing/leaving:** `remove_user_from_organization` deletes all active memberships and access-control docs for the target user in that organization, rejects removing the owner, lets a non-owner member remove only themself as an organization leave action, and requires `organization.members.manage` to remove another user.
- **Notifications:** invite notifications are in-app only in v1; no outbound email is sent. Backend notification listing treats `notifications` docs as the source of truth and lists the current user's docs without rechecking organization membership. `workspaceId` is retained for display/navigation when the originally invited workspace still exists; if that workspace is gone, the UI opens the organization primary/home workspace. Organization member removal, workspace deletion, and organization deletion delete affected invite notifications so invalid docs do not stay visible.

# Ownership transfer

- `access_control.transfer_organization_ownership` is owner-only and rejects the default organization.
- The new owner must be an active member of the organization `home` workspace.
- The new owner must have an available `extra_organizations` quota slot. Transfer releases one old-owner quota unit and consumes one new-owner quota unit.
- The old owner remains a regular member through existing memberships and a default-workspace `member` assignment unless a separate flow removes them.
- Owner-billed organizations automatically bill the new owner for operations started after transfer because billing resolves `organizations.ownerUserId` at operation start.

# Active memberships

- **Field:** `organizations_workspaces_users.active` — required boolean. `false` only during account-deletion retention so docs stay for recovery but are non-effective; normal docs store `active: true`.
- **Indexes:** `by_workspace_user_active`, `by_user_organization_workspace_active`, `by_active_organization_workspace_user`, `by_active_user_organization_workspace` — prefix with `eq("active", true)` so hot paths avoid post-query filtering.

# Creating extra workspaces

- **`create_workspace`** is allowed in the **default** organization as well as others (membership check: user must already have at least one membership in that organization). Extra workspaces are **not** the primary workspace unless created by `organizations_db_create`.

# Organization and workspace deletion and data purge

**Phase 1 — UI-facing / structural (immediate)** | **Phase 2 — Heavy content (one cron)**

| Entrypoint                                                                                                                   | Phase 1                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            | Queue doc                                                                                                                                                                                                                                                                                                    | Phase 2                                                                                                                                                                                                                                                                                                                             |
| ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `organizations.delete_workspace`                                                                                             | Queue purge + delete all memberships on that workspace + delete `organizations_workspaces` doc + release one organization `extra_workspaces` quota unit                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            | One `data_deletion_requests` doc with `scope: "workspace"` plus `userId`, `organizationId`, `workspaceId`, and purge eligibility                                                                                                                                                                             | `process_workspace_deletion_request` deletes one limited tenant-content batch for `(organizationId, workspaceId)` per call, keeps the queue doc while work remains, and removes it only after all covered content is gone                                                                                                           |
| `organizations.delete_organization`                                                                                          | Owner-only. Queue one doc (`scope: "organization"`, `organizationId` only); delete all **memberships** and access-control docs immediately; release one owner `extra_organizations` quota unit; **defer** deleting `organizations`, `organizations_workspaces`, and organization quota docs until the worker                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       | One `data_deletion_requests` doc with `scope: "organization"` plus `userId`, `organizationId`, and purge eligibility                                                                                                                                                                                         | `process_organization_deletion_request` processes queued workspace requests in that organization (even if a workspace doc was already removed), then existing `organizations_workspaces`, then organization structural docs in limited retryable batches                                                                            |
| `access_control.transfer_organization_ownership` / `organizations.delete_organization` + `users.delete_current_user_account` | Account management calls `users.list_current_user_account_deletion_blocking_organizations` as a preflight and opens one resolver modal for owned non-personal organizations. Each doc links to `/w/{organization}/{home}/users` for the regular Users route ownership-transfer flow, or lets the user confirm deleting the organization through `organizations.delete_organization`. The user-facing `users.delete_current_user_account` repeats the same `organizations.ownerUserId` blocker check, ignores organizations already queued with a `scope: "organization"` deletion request, and refuses deletion until every remaining blocker has been transferred or deleted. After blockers are gone, the user tombstone creates/reuses one `scope: "user"` doc, sets `users.deletedAt`, and sets `active: false` on remaining memberships. `internal.data_deletion.init_user_deletion` still queues still-owned non-personal organizations for deletion when called directly by internal/admin lifecycle paths. | One `scope: "user"` doc for normal user-facing account deletion; resolver organization-delete docs create their own `scope: "organization"` docs through `organizations.delete_organization`; internal/admin direct initializer calls may also create organization docs for still-owned organization cleanup | Eligible at `eligibleAt`: `process_user_deletion_request` finalizes user-owned state, deletes remaining access-control docs for the deleted user, and queues now-eligible organization purge docs for organizations that have no active users left; shared transferred organizations remain untouched when active users still exist |

Organization deletion requests are retryable and may outlive some structure docs. The worker deletes by the request organization id, handles queued workspace docs for workspaces whose `organizations_workspaces` doc is already gone, clears matching scoped cleanup docs such as quotas, and removes the organization queue doc only when the organization doc and related structure are gone.

Workspace-scope queue docs are content-only after `organizations.delete_workspace`, because that UI-facing phase already removed memberships, access-control docs, quota usage, and the workspace doc. Internal full-delete flows that still have a workspace doc use `db_delete_workspace_batch` to delete content first and workspace structure after that.

**Unified cron:** [crons.ts](../../../packages/app/convex/crons.ts) runs **`data_deletion.enqueue_deletion_requests_processing`** daily. The action enqueues the `data_deletion_workpool` Workpool worker (`process_deletion_requests`), which processes eligible `user` requests first, then `organization`, then `workspace`, with a limited mutation-step budget per worker run. If eligible work remains, the worker enqueues a successor Workpool action instead of letting one run grow without a cap.

**Auth identity:** `server_convex_get_user_fallback_to_anonymous` only reads the JWT; it does not load `users` or gate on `deletedAt`. This is a current security gap for tombstoned anonymous users. Account-level anonymous writes must load a live user row, but current handlers do not all do so. See [auth-system: Known anonymous deletion gap](../auth-system/SKILL.md#known-anonymous-deletion-gap) before changing an anonymous tenant write.

## Data-only account reset

`users.hard_delete_user_now` with omitted `purgeUserMod` or `"data"` is a data-only account reset, not account deletion. It preserves the auth-capable `users` doc, profile, billing state, default `personal` organization, and default `home` workspace. It cancels the user-scope deletion request, but keeps resource-scope queue docs until the reset either consumes them or clears the docs that target the preserved default organization/workspace. It uses limited organization/workspace purge batches and schedules the same user/mode when more reset work remains. Auth-removing modes tombstone and drain user sessions before external cleanup, then enqueue the Workpool once for any organization/workspace requests; they never process the global queue inline. Content inside `home` is purged while the default organization/workspace docs stay usable, extra workspaces under `personal` are deleted, owned non-default organizations are deleted only when no other active user belongs to that organization, and extra shared workspaces are deleted only when the reset user is the sole active member of that workspace. Auth-removing finalization transfers a surviving shared organization to an active default-workspace member before removing its owner. Missing or inconsistent default tenant docs are treated as invariant errors, not repair cases.

## Content purge coverage (`process_organization_deletion_request` / `process_workspace_deletion_request`)

This purge includes pending-update state; AI files, threads, messages, and thread state; public API credentials, grants, and write stages; plugin runs, handlers, installations, secrets, and UI sessions; chat messages; file metadata, chunks, Yjs state, snapshots, stats, materialization jobs, R2 assets and objects; and file nodes last. Treat [data-deletion: Workspace Content Purge Coverage](../data-deletion/SKILL.md#workspace-content-purge-coverage) as the ordered list. It also records the current `activities` cleanup gap; do not describe the purge as exhaustive until that gap is fixed.

**Not present in Convex schema:** there is no `human_thread_messages` table; comments/human threads are not a separate purge target in this codebase today.

**Scale note:** Workspace content purge must use indexed limited reads (`take(batchSize)`) or one-parent-at-a-time child deletion. Do not reintroduce tenant-sized `.collect()` reads in purge paths.

## Queue table (summary)

| Table                    | Purpose                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `data_deletion_requests` | Shared delayed deletion queue. `scope` means what is being deleted: `workspace` \| `organization` \| `user`. `userId` is always required; `organizationId`/`workspaceId` are present for organization/workspace docs. Docs always set `eligibleAt`. Account deletion always creates/reuses a `user` scope doc and may also create now-eligible `organization` scope docs for still-owned or newly-empty organizations. |

# Resolution helpers

- **`get_membership_by_organization_workspace_name`:** resolves validated names against the **current user’s** membership docs and returns only the matching membership doc; **first matching** organization+workspace pair wins (no global sort of candidates). UI that needs organization metadata, including `organization.default`, should compose it from `organizations.list` instead of carrying it in tenant context.
- **`list`:** sorts organizations (default first) and workspaces (primary first, then name / id). Organization docs include `ownerUserId`, so UI can label owner-billed organizations without querying billing state.

## Purge worker index rule

- Prefer **narrow index reads** per `(organizationId, workspaceId)` and eligible queue docs by due time. If a new tenant-scoped table can grow, add the purge index when adding the table or include a parent-doc batching strategy that never collects the whole tenant.

## Account-deletion retention TODO

- Today, phase 2 still runs after the fixed retention window even if a paid subscription was only scheduled to end at billing-period close.
- In the future, once long-running plans such as yearly subscriptions exist, phase 2 should wait until subscription end when that is later than retention so paid users do not lose their data before the paid term finishes.
